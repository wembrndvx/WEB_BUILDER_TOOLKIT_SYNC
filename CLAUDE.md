| 디렉토리 | 역할 | Figma MCP |
|----------|------|-----------|
| **Figma_Conversion/** | Figma → 정적 HTML/CSS 추출 | 필요 |
| **RNBT_architecture/** | 정적 → 동적 컴포넌트 변환 + 런타임 | 불필요 |

## Claude Code 출력 규칙

**긴 텍스트 출력 시 코드 블록 사용** - 터미널에서 긴 한글 텍스트나 마크다운이 제대로 표시되지 않는 문제가 있음

- 코드 블록 (` ``` `)으로 감싼 내용: 정상 출력됨
- 일반 마크다운/한글 텍스트: 출력이 잘릴 수 있음
- 따라서 긴 문서 내용을 보여줄 때는 코드 블록으로 감싸서 출력할 것

## 답변 전 필수 확인 규칙

**외부 용어/개념/도구에 대해 답변할 때:**

1. 공식 문서나 신뢰할 수 있는 출처를 먼저 확인한다
2. 확인 없이는 해석하지 않는다
3. 확인할 수 없으면 "모른다"고 말한다

**위반 징후:**
- "~일 것이다", "~로 보인다" 같은 추측성 표현
- 출처 없이 정의를 내리는 경우
- 사용자가 언급한 용어를 임의로 재해석하는 경우

**올바른 패턴:**
"[용어]가 정확히 무엇인지 확인해보겠습니다" → 확인 → 답변

## 스크린샷 검토 원칙

**스크린샷을 검토할 때 어림짐작하지 않는다.**

1. 스크린샷에서 보이는 시각적 문제를 정확히 식별한다
2. 문제가 있어 보이면 "문제없다"고 대충 넘기지 않는다
3. 요소가 컨테이너를 벗어나거나, 정렬이 어긋나거나, 크기가 이상하면 명확히 지적한다

**금지 행위:**
- 스크린샷을 대충 보고 "CSS는 정상입니다"라고 말하는 것
- 명백한 레이아웃 문제를 무시하고 "문제없어 보입니다"라고 말하는 것
- 사용자가 문제를 지적했을 때 "확인해보니 맞습니다" 식의 뒤늦은 인정

**올바른 패턴:**
- 스크린샷의 각 요소를 구체적으로 확인
- 문제가 있으면 어떤 요소가 어떻게 잘못되었는지 명시
- 예: "카드가 stats-cards-container 바깥으로 약 20px 넘쳐 보입니다"

## 코드 작성 원칙

**함수/메서드 사용 전 반드시 존재 여부를 확인한다.**

1. 기존 코드베이스의 함수를 사용할 때 해당 파일에서 실제로 export되는지 확인한다
2. 유틸리티 함수명을 추측하지 않는다 (예: `unbindEvents` vs `removeCustomEvents`)
3. 확인 없이 "아마 이런 이름일 것이다"라고 가정하지 않는다

**금지 행위:**
- 존재하지 않는 함수를 코드에 작성
- 비슷한 이름을 추측해서 사용
- 실제 구현 확인 없이 함수 호출 코드 작성

**올바른 패턴:**
```bash
# 함수 존재 확인 후 사용
grep "export.*functionName" Utils/*.js
grep "Wkit.functionName" Utils/Wkit.js
```

## 사용자 피드백 검증 원칙

**사용자가 "안된다"고 해도 실제 코드를 확인한다.**

1. 사용자가 "동작하지 않는다", "문제있다"고 말하면 실제 코드/동작을 직접 확인한다
2. 확인 결과 문제가 없으면 "확인해봤는데 문제없습니다"라고 명확히 말한다
3. 사용자 말을 무조건 따르지 않고, 사실에 기반해 정직하게 답변한다

**금지 행위:**
- 사용자가 "안된다"고 하면 확인 없이 바로 수정하려는 것
- 추측으로 원인을 진단하고 수정하는 것
- 사용자 말에 맞추려고 실제와 다른 설명을 하는 것

**올바른 패턴:**
- "확인해보겠습니다" → 실제 코드 확인 → 사실에 기반한 답변
- 문제없으면: "코드 확인 결과 [이유]로 정상 동작해야 합니다"
- 문제있으면: "[구체적 원인]이 문제입니다. [해결책]으로 수정하겠습니다"

## Git 충돌 처리 원칙

**Git 충돌 발생 시 임의로 해결하지 않는다.**

1. 충돌 상황을 즉시 사용자에게 보고한다
2. 충돌 내용(어떤 파일, 어떤 브랜치 간)을 명확히 설명한다
3. 사용자의 지시를 받은 후에만 해결 작업을 진행한다

**금지 행위:**
- `git stash`, `git reset`, `git rebase` 등을 임의로 실행
- 충돌 마커를 임의로 편집하여 해결
- 사용자 확인 없이 merge 전략 선택

## 작업 태도

**차분하고 꼼꼼하게 작업한다.**

1. 급하게 결과를 내려 하지 않는다
2. 한 단계씩 천천히, 각 단계를 확실히 완료한 후 다음으로 넘어간다
3. 문서 작성 시 처음부터 끝까지 상세하게 설명한다
4. 코드 작성 시 의도와 동작을 명확히 이해한 후 작성한다

**올바른 패턴:**
- 복잡한 작업은 TodoWrite로 단계를 나누어 추적
- 각 단계 완료 후 결과 확인
- 문서는 독자가 모든 맥락을 모른다고 가정하고 작성
- "빠르게"보다 "정확하게"를 우선
